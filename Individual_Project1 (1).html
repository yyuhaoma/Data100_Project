Individual Planning Report

1. Data Description

The dataset consists of two files: players.csv and sessions.csv.

Players Dataset (players.csv)

  • Number of observations: 196 players
  • Number of variables: 7
  • Variable Summary:
Variable Type Description
experience Categorical Player experience level (e.g., Beginner, Amateur, Pro)
subscribe Boolean Whether the player subscribed to the game-related newsletter
hashedEmail String Unique identifier for each player (anonymized)
played_hours Numeric Total number of hours the player has played
name String Player's name (not relevant for analysis)
gender Categorical Player's gender
Age Numeric Player's age (with some missing values)

Sessions Dataset (sessions.csv)

  • Number of observations: 1535 player sessions
  • Number of variables: 5
  • Variable Summary:
Variable Type Description
hashedEmail String Unique identifier for linking to players.csv
start_time String Start time of the session
end_time String End time of the session (2 missing values)
original_start_time Numeric Unix timestamp of start_time
original_end_time Numeric Unix timestamp of end_time

Issues Identified:

  • sessions.csv contains missing values in end_time, which could affect the accuracy of session duration calculations and the identification of peak player activity periods.
  • start_time and end_time need to be properly formatted and converted into structured time-based features (hour, weekday, month) to facilitate meaningful trend analysis.
  • Some session timestamps may be recorded in different formats or time zones, requiring standardization to ensure consistency.
  • Aggregating player activity across different time frames (hourly, weekly, monthly) introduces variability, making it essential to handle potential data gaps and inconsistencies.
  • The presence of extreme session lengths (e.g., excessively long or short play sessions) could skew the analysis and may require filtering or normalization.

2. Questions

Broad Question:

What time windows have the highest number of active players?

Specific Question:

Can session start and end times from sessions.csv be used to accurately determine peak gaming hours, active weekdays, and high-engagement months?

How the data helps:

The start_time and end_time variables enable us to compute the number of active players at any given moment.
By aggregating session data across hourly, weekly, and monthly time frames, we can uncover patterns in player activity and identify peak usage periods.

4. Methods and Plan

The objective of this study is to forecast player demand, specifically predicting the number of concurrent players on the server at any given time. To achieve this, we propose using the K-Nearest Neighbors (KNN) model, which utilizes historical player activity to estimate future concurrent player counts.

Why choose KNN?

  • Captures time-dependent gaming patterns: KNN predicts player activity by identifying past time periods with similar player engagement levels, making it well-suited for modeling cyclical and recurring trends.
  • Effective for structured time-series data: Since server logs provide detailed timestamps, KNN can leverage this structured format to make reliable predictions without requiring complex feature engineering.

Assumptions

  • Historical player trends are predictive of future patterns, meaning similar time windows will likely have correlated player activity levels.
  • Server logs are accurate, with correctly recorded start_time and end_time, ensuring precise modeling of session durations.
  • External factors such as holidays and in-game events are either reflected in the data or have minimal impact on the overall trend.
  • Server performance remains stable, with no unexpected downtimes or anomalies that could distort the prediction model.

Potential Limitations

  • KNN is sensitive to outliers, meaning that extreme fluctuations, such as unexpected server failures or major game updates, could negatively impact prediction accuracy.
  • Gaps in the dataset prevent full seasonal analysis, as missing months mean KNN cannot generalize for unseen time periods, potentially distorting long-term trend forecasting.

Data Processing Plan

  1. Transform timestamps into structured time features: Extract relevant variables such as hour, weekday, and month to create a feature set based on time windows.
  2. Calculate concurrent player counts: For each time point, compute the number of active players by counting sessions where start_time is earlier than the given time and end_time is later. This serves as the target variable for the model.
  3. Optimize KNN hyperparameters: Use cross-validation to determine the optimal value of k, ensuring a balance between overfitting and underfitting.
  4. Evaluate model performance: Measure prediction accuracy using Root Mean Squared Percentage Error (RMSPE) to assess error margins and overall model reliability.

Conclusion

This report aims to forecast player demand by predicting the number of concurrent players on the server at any given time. Through exploratory data analysis, we identified that peak activity occurs around 4 AM, weekends see the highest engagement, and June has the highest recorded sessions among the available data.

To achieve accurate demand forecasting, we propose using the K-Nearest Neighbors (KNN) model, which leverages historical time-windowed player activity to predict future trends. This method is well-suited for structured datasets and effectively captures recurring gaming patterns.

While KNN is a robust approach, outliers, missing data, and seasonal variations present challenges that must be accounted for. To mitigate these limitations, cross-validation and RMSPE evaluation will be used to optimize model performance and ensure reliability.

By accurately forecasting player demand, this study can help optimize resource allocation, server stability, and recruitment strategies, ultimately improving the overall player experience. Future work could explore additional predictive methods, such as time-series models, to enhance forecasting precision.
